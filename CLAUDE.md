# MorseWave — контекст для Claude Code

Тренажёр азбуки Морзе. PHP (без фреймворка) + MySQL (PDO) + ванильный JS.
Живёт на https://morse.r9old.ru, репозиторий: https://github.com/loloka/morsewave

Полное описание проекта, история изменений и архитектура — в [README.md](README.md).
Этот файл — только то, что нужно знать *перед тем как писать код*.

## Локальный запуск

```bash
docker compose up -d --build
```
Сайт: http://localhost:8080 · phpMyAdmin: http://localhost:8081 (root / morsewave_root)

При первом запуске MySQL-контейнер автоматически накатывает `database/schema.sql`
и все `database/migration_*.sql` по порядку (см. `docker-compose.yml`,
`docker-entrypoint-initdb.d`). Это происходит **только один раз**, на пустом
volume. Если база уже существует и нужно применить новую миграцию — накатывай
её вручную через phpMyAdmin или `docker compose exec db mysql ...`.

## Обязательные правила (не откатывать без явной просьбы пользователя)

- **XP — всегда целые числа.** `Progress.addXp()` в `assets/js/progress.js`
  округляет через `Math.round`. Если формула даёт дробный множитель —
  округляй результат ДО вызова `addXp`, не после. Повторное сложение
  дробных чисел в JS даёт ошибку плавающей точки (`592.5000000000002`) —
  это уже было багом, не повторяй его.
- **Один `AudioContext` на весь сайт.** Никогда не создавай
  `new AudioContext()` напрямую — используй `getSharedAudioContext()` из
  `assets/js/audio.js`. Причина: у браузеров есть лимит на число
  одновременно живых контекстов; создание нового на каждое воспроизведение
  рано или поздно приводило к зависанию тренировки без звука и без реакции
  на клики (было багом, чинили специально). Все `await audio.play(...)`
  оборачивай в `try/catch/finally`, сбрасывающий флаг занятости
  (`isPlaying`/`recBusy`) в `finally`.
- **wpm-контролы — везде `<input type="range">` с шагом 1**, не `<select>`
  с пресетами. Осознанное решение пользователя.
- **XP не должен фармиться** повторением тривиального действия. При
  добавлении нового источника XP — закладывай защиту (см. `learn.js`: XP
  только пока символ/условие не «закрыто»; см. `groups.js`: ставка XP
  зависит от размера набора символов и длины группы).
- **Схема БД меняется только через миграции.** Не редактируй `schema.sql`
  молча — если меняешь структуру/данные, добавляй новый
  `database/migration_N.sql` (следующий номер по порядку), обнови и
  `schema.sql` (чтобы свежие установки были согласованы), и `docker-compose.yml`
  (добавь монтирование нового файла в `db.volumes`), и README (структура
  проекта, список миграций в разделе деплоя, changelog).
- **CLI-скрипты (`database/seed_callsigns.php` и подобные) — без стрелочных
  функций (`fn`) и без строгой типизации параметров.** На проде (SSH-консоль
  хостинга) может стоять более старая версия PHP, чем PHP-FPM самого сайта —
  уже ловили из-за этого `Call to undefined function fn()`.
- **Сигнальная линия и лампа — сквозной элемент дизайна** (`SignalLine`,
  `MorseLamp` из `assets/js/signal.js`). Новые режимы тренировки должны их
  подключать. Видимость сигнальной линии переключается через
  `wireSignalVisibilityToggle()` и общую настройку `DisplaySettings` — это
  сознательно (подсказка-читерство при тренировке слуха), не убирай
  переключатель.
- Пользователь развёртывает на VPS (FastPanel, PHP 8.4) через `git pull`.
  Продовые пути `config/`, `database/` и `includes/` закрыты `.htaccess` —
  не убирай эту защиту (там креды от БД, SQL-файлы и логика сессий).
- **Аккаунты — надстройка, не источник истины.** Прогресс пользователя
  (`Progress` в `assets/js/progress.js`) живёт в `localStorage`. Таблицы
  `users`/`user_stats` — только публичный слепок XP/серии для лидерборда
  на главной. Не переписывай логику так, будто сервер — единственный
  источник прогресса; localStorage должен продолжать работать полностью
  автономно даже без аккаунта.
- **Почта — через Resend API** (`includes/resend_mailer.php`,
  `config/mail.php`), не через SMTP хостинга. Был реальный кейс: у
  `mail.r9old.ru` не настроен DKIM, а PTR общего IP не связан с доменом —
  Gmail тихо резал письма без bounce. Не откатывай на SMTP-подход без явной
  причины — уже проверено, что не работает надёжно на этом хостинге.

## Открытые задачи

### Полная синхронизация прогресса между устройствами (не начато)

Сейчас (см. правило «Аккаунты — надстройка» выше) на сервер публикуется
только слепок для лидерборда (`xp`, `streak_count`, `streak_last_date` —
таблица `user_stats`, эндпоинт `api/sync_progress.php`), и то **только по
явному клику «Опубликовать»** на странице профиля (не автоматически — см.
запись в changelog про переход на opt-in-публикацию). Это осознанно
**отдельная** система от задачи ниже — не путать одно с другим:

- **Публикация в лидерборд** — по-прежнему только вручную, кнопкой.
  Публичное и опциональное.
- **Полная синхронизация прогресса** (эта задача) — весь `Progress`-объект
  целиком (не только xp/streak), приватно, для того чтобы человек мог
  зайти с телефона и продолжить с того места, где остановился на
  компьютере. Не публичная витрина, а личное хранилище — на будущее можно
  даже сделать так, что оно не требует подтверждённой почты (в отличие от
  публикации в лидерборд, где верификация — барьер против дублей).

**План реализации:**

1. Новая таблица (`database/migration_10.sql` + обновить `schema.sql` +
   примонтировать в `docker-compose.yml`):
   ```sql
   CREATE TABLE user_progress (
       user_id      INT PRIMARY KEY,
       progress_json LONGTEXT NOT NULL,
       updated_at   DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
       FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
   );
   ```
   Простой JSON-блоб, не пытаться раскладывать `Progress` по колонкам —
   структура состояния уже сложная (вложенные объекты/массивы) и
   продолжит меняться, реляционная схема под неё — лишняя работа.

2. `api/push_progress.php` (POST, требует логин) — сохраняет весь
   `JSON.stringify(Progress.load())` в `user_progress`. Вызывать из
   `progress.js` при значимых изменениях (по аналогии с текущим
   `syncToServer`, но шлёт **весь** объект, не только xp/streak) — с
   debounce, как сейчас.

3. `api/pull_progress.php` (GET, требует логин) — отдаёт сохранённый
   `progress_json`.

4. **Слияние при логине** — самое сложное место. Нельзя просто
   перезаписывать localStorage данными с сервера (потеряется то, что
   успело накопиться в этом браузере с последней синхронизации) и нельзя
   просто игнорировать сервер. Стратегия: слияние по каждому полю
   отдельно, никогда не выбирая меньшее:
   - `xp`, `streak.count`, `kochLevel`, все числовые счётчики в `stats` — `Math.max(локальное, серверное)`
   - `learnedLetters`, `unlockedAchievements` — объединение множеств (union)
   - `streak.lastDate` — более поздняя дата
   Реализовать как `Progress.mergeFromServer(serverState)` в `progress.js`,
   вызывать сразу после успешного логина (`api/login.php` /
   `api/register.php` ответ) — тихо, без диалогов подтверждения: raw-merge
   по максимуму гарантированно не теряет прогресс ни с одной стороны, поэтому
   спрашивать пользователя не нужно.

5. После слияния — сразу вызвать `push_progress`, чтобы результат слияния
   попал на сервер (иначе следующий логин на первом устройстве откатит
   мерж).

**Не забыть**: если пользователь жмёт «Сбросить весь прогресс» в
«Достижениях» — `api/push_progress.php` тоже должен обнулиться (или
удалиться), не только `api/unpublish_stats.php` для лидерборда (см. как
это уже сделано для лидерборда — используй как образец).
