# MorseWave — контекст для Claude Code

Тренажёр азбуки Морзе. PHP (без фреймворка) + MySQL (PDO) + ванильный JS.
Живёт на https://morse.r9old.ru, репозиторий: https://github.com/loloka/morsewave

Полное описание проекта и архитектура — в [README.md](README.md),
история изменений — в [CHANGELOG.md](CHANGELOG.md).
Этот файл — только то, что нужно знать *перед тем как писать код*.

## Локальный запуск

```bash
docker compose up -d --build
```
Сайт: http://localhost:8080 · phpMyAdmin: http://localhost:8081 (root / morsewave_root)

При первом запуске MySQL-контейнер автоматически накатывает
`database/schema.sql` — полную актуальную схему с сидами (см.
`docker-compose.yml`, `docker-entrypoint-initdb.d`). Это происходит
**только один раз**, на пустом volume. На уже существующей базе изменения
схемы применяются вручную через phpMyAdmin или `docker compose exec db mysql ...`.

## Обязательные правила (не откатывать без явной просьбы пользователя)

- **XP — всегда целые числа.** `Progress.addXp()` в `assets/js/progress.js`
  округляет через `Math.round`. Если формула даёт дробный множитель —
  округляй результат ДО вызова `addXp`, не после. Повторное сложение
  дробных чисел в JS даёт ошибку плавающей точки (`592.5000000000002`) —
  это уже было багом, не повторяй его.
- **Один `AudioContext` на весь сайт.** Никогда не создавай
  `new AudioContext()` напрямую — используй `getSharedAudioContext()` из
  `assets/js/audio.js`. Причина: у браузеров есть лимит на число
  одновременно живых контекстов; создание нового на каждое воспроизведение
  рано или поздно приводило к зависанию тренировки без звука и без реакции
  на клики (было багом, чинили специально). Все `await audio.play(...)`
  оборачивай в `try/catch/finally`, сбрасывающий флаг занятости
  (`isPlaying`/`recBusy`) в `finally`.
- **wpm-контролы — везде `<input type="range">` с шагом 1**, не `<select>`
  с пресетами. Осознанное решение пользователя.
- **XP не должен фармиться** повторением тривиального действия. При
  добавлении нового источника XP — закладывай защиту (см. `learn.js`: XP
  только пока символ/условие не «закрыто»; см. `groups.js`: ставка XP
  зависит от размера набора символов и длины группы).
- **`database/schema.sql` — единственный источник истины по схеме БД.**
  Файлы миграций консолидированы в него и удалены в v2.28 (история — в
  git). Меняешь структуру или сид-данные: (1) правь `schema.sql` — свежие
  установки должны получать актуальную базу одним файлом; (2) в записи
  CHANGELOG.md приведи готовые ALTER/UPDATE/INSERT-команды для уже
  работающей базы — владелец живого сервера выполняет их вручную после
  `git pull` (автоматически initdb работает только на пустом volume);
  (3) новые файлы `migration_N.sql` больше НЕ создавай.
- **CLI-скрипты (`database/seed_callsigns.php` и подобные) — без стрелочных
  функций (`fn`) и без строгой типизации параметров.** На проде (SSH-консоль
  хостинга) может стоять более старая версия PHP, чем PHP-FPM самого сайта —
  уже ловили из-за этого `Call to undefined function fn()`.
- **Сигнальная линия и лампа — сквозной элемент дизайна** (`SignalLine`,
  `MorseLamp` из `assets/js/signal.js`). Новые режимы тренировки должны их
  подключать. Видимость сигнальной линии переключается через
  `wireSignalVisibilityToggle()` и общую настройку `DisplaySettings` — это
  сознательно (подсказка-читерство при тренировке слуха), не убирай
  переключатель.
- Пользователь развёртывает на VPS (FastPanel, PHP 8.4) через `git pull`.
  Продовые пути `config/`, `database/` и `includes/` закрыты `.htaccess` —
  не убирай эту защиту (там креды от БД, SQL-файлы и логика сессий).
- **Аккаунты — надстройка, не источник истины.** Прогресс пользователя
  (`Progress` в `assets/js/progress.js`) живёт в `localStorage`. Таблицы
  `users`/`user_stats` — только публичный слепок XP/серии для лидерборда
  на главной. Не переписывай логику так, будто сервер — единственный
  источник прогресса; localStorage должен продолжать работать полностью
  автономно даже без аккаунта.
- **Слияние прогресса при логине — только «вверх»** (`Progress.mergeFromServer`
  в `assets/js/progress.js`): числа — `Math.max`, множества
  (`learnedLetters`, `unlockedAchievements`) — union, даты — более поздняя.
  Никогда не выбирать меньшее и не перезаписывать одну сторону другой —
  raw-merge по максимуму гарантирует, что прогресс не теряется ни с одной
  стороны, поэтому диалоги подтверждения не нужны. Приватная синхронизация
  (`user_progress`, `api/push_progress.php`/`pull_progress.php`) и публичный
  лидерборд (`user_stats`, кнопка «Опубликовать») — две РАЗНЫЕ системы, не
  смешивать: синхронизация не требует подтверждённой почты и работает тихо,
  публикация в лидерборд — только вручную и только с подтверждённой почтой.
  «Сбросить весь прогресс» обязан удалять и серверную копию
  (`api/delete_progress.php`), иначе следующий логин откатит сброс.
- **Любая серверная проверка пароля — через общий счётчик блокировки**
  (`failed_login_attempts`/`locked_until`, лимит 5 попыток / 15 минут, как в
  `api/login.php`). Если добавляешь новый эндпоинт, принимающий пароль
  (сейчас это login и `update_account.php`), — он обязан и уважать активную
  блокировку, и инкрементировать счётчик при неверном пароле, иначе станет
  обходным путём для брутфорса мимо защиты логина.
- **Бонус задания дня требует точность ≥ 50 %**, не только совпадение
  параметров сессии — иначе бонус фармится прокликиванием пустых ответов
  (был реальный баг, закрыт в v2.26). Обычный XP за верные символы при
  низкой точности остаётся.
- **Почта — через Resend API** (`includes/resend_mailer.php`,
  `config/mail.php`), не через SMTP хостинга. Был реальный кейс: у
  `mail.r9old.ru` не настроен DKIM, а PTR общего IP не связан с доменом —
  Gmail тихо резал письма без bounce. Не откатывай на SMTP-подход без явной
  причины — уже проверено, что не работает надёжно на этом хостинге.

## Открытые задачи (бэклог — согласован с владельцем, порядок ≈ приоритет)

Браться по возможности, сверху вниз. Перед реализацией перечитай
«Обязательные правила» выше — особенно про XP, AudioContext и schema.sql.

1. **Индикатор синхронизации** в профиле: метка вида «Синхронизировано
   только что / 5 мин назад» — push сейчас тихий, и человек не знает,
   сработал ли он. Данные уже есть (`updated_at` в `api/pull_progress.php`,
   момент успешного `pushNow` можно запоминать на клиенте).
2. **Экспорт/импорт прогресса файлом** (страховка для тех, кто без
   аккаунта): «Скачать бэкап» → JSON из `Progress.load()`, «Загрузить» →
   через существующий `Progress.mergeFromServer()` (сливает «только
   вверх», ничего не потеряется). ВАЖНО: интерфейс минимальный и понятный
   всем поколениям радиолюбителей — две кнопки, без нагромождения в
   профиле.
3. **Тренировка на реальных словах**: топ частых слов + радиообменные
   фразы на приём. Размещение: раздел «Группы» (`groups.php`), третьим
   табом наравне с «Группы символов» и «Сокращения».
4. **Кириллическая азбука Морзе** (спрашивали пользователи!):
   - «Буквы → отправка ключом»: кириллица как третий набор наравне с
     «Алфавит» и «Порядок Коха».
   - «Буквы → приём на слух»: отдельный набор «Кириллические символы».
   - XP за кириллицу чуть выше — символы объективно сложнее (длиннее коды).
   - КРИТИЧНО: не сломать существующий прогресс. Латинская A и
     кириллическая А — РАЗНЫЕ символы в `learnedLetters` (нужен префикс
     или отдельное поле), ачивка `full_alphabet` (36) считается только по
     латинице+цифрам, merge при синхронизации должен переживать состояние
     без кириллических полей (union это уже умеет).
5. **Мини-игра в стиле Марио** (владелец очень ждёт): платформер-раннер,
   где препятствия проходятся выбором правильного символа на слух и/или
   удержанием правильного ритма ключа (точка — короткий прыжок, тире —
   длинный?). Отдельный пункт меню (место в таббаре освобождено в v2.24).
   Механику согласовать с владельцем перед реализацией — начать с
   короткого описания вариантов геймплея.
6. **Оценка ритма ключа** — НЕ для новичков, а как третий подрежим в
   «Буквах» наравне с «Отправка ключом» и «Приём на слух»: показывать
   отклонение длительностей от идеала (точка:тире:паузы = 1:3:1,7).
   XP — соразмерно сложности. Наработки пригодятся и для мини-игры (п.5).
7. **Ссылка на MorseWalker** — родственный проект владельца
   (https://github.com/loloka/morsewalker, связаны через сайт r9o.ru),
   QSO-симулятор. Упомянуть в подходящем месте (футер? главная, рядом с
   «путём новичка» как следующий шаг после тренажёра?). Согласовать место.
8. **(Дальний прицел) Полноценное Android-приложение.** НЕ PWA — владелец
   явно против («выглядит неполноценно»). Отдельный большой проект.
9. **(Исследование) Подключение реального телеграфного ключа.** Владельцу
   интересно на будущее. Реалистичные пути: (а) самый простой — адаптер
   на Arduino/Pro Micro, эмулирующий клавиатуру (замыкание ключа =
   нажатие пробела; наш ввод в `input.js` уже слушает пробел — заработает
   без правок кода); (б) Web Serial API для самодельных адаптеров;
   (в) микрофонный вход + детекция тона через Web Audio (ключ радиостанции
   с сайдтоном). Начать с (а) — возможно, достаточно просто написать
   инструкцию-страничку, как собрать адаптер.

Отклонено владельцем (не предлагать повторно без его инициативы):
- QSO-симулятор — уже есть отдельный проект MorseWalker (см. п.7).
- PWA — владелец не любит, ждём полноценный Android (п.8).
- График XP по дням — «лишнее нагромождение».
- Недельный лидерборд — пока не нужен, вечная таблица мотивирует.

## Прежде чем коммитить

- Проверь синтаксис изменённых `.php` файлов (`php -l путь/к/файлу.php`,
  доступно внутри контейнера: `docker compose exec app php -l ...`).
- Добавь короткую запись в CHANGELOG.md (свежие записи сверху) — так
  следующая сессия (и человеческая, и ИИ) видит, что менялось.
